# NamespaceValidationAutoFix

Este proyecto incluye una validacion MSBuild para comprobar que el `namespace` de cada archivo `.cs` coincide con su ruta dentro del proyecto.

La regla vive en:

```text
Directory.Build.targets
build/ValidateNamespaceMatchesFolder.ps1
```

Como `Directory.Build.targets` se importa automaticamente por MSBuild, funciona tanto con:

```powershell
dotnet build
```

como con **Build Solution** en Visual Studio.

## Que valida

El validador calcula el namespace esperado usando:

```text
RootNamespace + carpeta relativa del archivo dentro del proyecto
```

Ejemplo:

```text
src/MVCPrueba1.Domain/Entities/PersonEntity.cs
```

Namespace esperado:

```csharp
namespace MVCPrueba1.Domain.Entities
```

Si el archivo contiene otro namespace, el build falla con `NS0001`.

## Modo normal

Por defecto, `NamespaceValidationAutoFix` esta desactivado:

```xml
<NamespaceValidationAutoFix Condition="'$(NamespaceValidationAutoFix)' == ''">false</NamespaceValidationAutoFix>
```

En este modo, el validador no modifica archivos. Solo falla el build y propone el namespace correcto.

Ejemplo de error:

```text
error NS0001: Entities\PersonEntity.cs namespace 'MVCPrueba1.Entities' does not match folder path. Suggested fix: replace it with 'namespace MVCPrueba1.Domain.Entities'.
```

## Activar autocorreccion desde consola

Para corregir namespaces automaticamente:

```powershell
dotnet build /p:NamespaceValidationAutoFix=true
```

Cuando esta activo, el validador cambia:

```csharp
namespace MVCPrueba1.Entities
```

por:

```csharp
namespace MVCPrueba1.Domain.Entities
```

## Activar autocorreccion en Visual Studio

Puedes activarlo temporalmente en `Directory.Build.targets`:

```xml
<PropertyGroup>
  <NamespaceValidationAutoFix>true</NamespaceValidationAutoFix>
</PropertyGroup>
```

Despues ejecuta **Build Solution**.

Recomendacion: vuelve a dejarlo en `false` cuando termines, para que el build normal no modifique archivos sin querer.

## Ejecutar desde External Tools en Visual Studio

Tambien puedes crear una herramienta externa para lanzar el autofix sin editar archivos MSBuild.

En Visual Studio:

1. Abre **Tools > External Tools...**.
2. Pulsa **Add**.
3. Configura la herramienta asi:

```text
Title:
Namespace Validation AutoFix

Command:
C:\Program Files\dotnet\dotnet.exe

Arguments:
build "$(SolutionDir)MVCPrueba1.slnx" /p:NamespaceValidationAutoFix=true

Initial directory:
$(SolutionDir)
```

4. Activa **Use Output window** si la opcion esta disponible.

Si Visual Studio muestra este error:

```text
The command is not a valid executable
```

no uses solo `dotnet` en `Command`; usa la ruta completa:

```text
C:\Program Files\dotnet\dotnet.exe
```

### Alternativa con cmd.exe

Si la ruta directa a `dotnet.exe` no funciona en tu instalacion, puedes configurar la herramienta usando `cmd.exe`:

```text
Title:
Namespace Validation AutoFix

Command:
C:\Windows\System32\cmd.exe

Arguments:
/c dotnet build "$(SolutionDir)MVCPrueba1.slnx" /p:NamespaceValidationAutoFix=true

Initial directory:
$(SolutionDir)
```

Esta alternativa abre `cmd.exe`, ejecuta el comando y devuelve la salida a Visual Studio.

## Que actualiza automaticamente

Cuando cambia un namespace antiguo por uno nuevo, tambien actualiza referencias en el repositorio:

```csharp
using MVCPrueba1.Entities;
```

pasa a:

```csharp
using MVCPrueba1.Domain.Entities;
```

Tambien actualiza:

```csharp
global using MVCPrueba1.Entities;
using static MVCPrueba1.Entities.SomeType;
```

y referencias fully-qualified dentro de texto, por ejemplo en migraciones de EF:

```csharp
modelBuilder.Entity("MVCPrueba1.Entities.PersonEntity", b => ...)
```

pasa a:

```csharp
modelBuilder.Entity("MVCPrueba1.Domain.Entities.PersonEntity", b => ...)
```

## Limpieza de using

Despues de actualizar namespaces, el script elimina `using` duplicados o redundantes.

Ejemplo:

```csharp
namespace MVCPrueba1.Application.UseCases.Persons
{
    using MVCPrueba1.Application.UseCases.Persons;
}
```

El `using` se elimina porque apunta al mismo namespace del archivo.

## Archivos generados

El validador no cambia la declaracion `namespace` de archivos generados como:

```text
*.g.cs
*.Designer.cs
// <auto-generated />
```

Pero en modo autocorreccion si puede actualizar sus `using` y referencias fully-qualified, porque esos archivos tambien participan en la compilacion.

## Desactivar la regla completa

Si necesitas desactivar la validacion para una compilacion concreta:

```powershell
dotnet build /p:EnforceNamespaceMatchesFolder=false
```

O en MSBuild:

```xml
<PropertyGroup>
  <EnforceNamespaceMatchesFolder>false</EnforceNamespaceMatchesFolder>
</PropertyGroup>
```
